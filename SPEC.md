# Fairfield County Events — Specification v0.1

## 1. Overview

A public website that aggregates as many Fairfield County, CT events as possible — town events, library programs, festivals, farmers markets, car shows, volunteer drives, meetups — into one browsable calendar. Users can submit events; an admin (Will) approves them before they appear.

**Non-goals (v1):** user accounts, ticket sales, professional/leisure categorization, paid data sources (SerpApi is a documented v2 add-on), Instagram ingestion.

## 2. Tech Stack

| Layer | Choice | Rationale |
|---|---|---|
| Backend | Python 3.11+, FastAPI | Async, typed, OpenAPI docs for free; Python is best-in-class for scraping (httpx, BeautifulSoup, icalendar) |
| Ingestion | Standalone Python jobs (`ingest/`) run on a scheduler (cron / APScheduler) | Decoupled from the web API; a broken scraper never takes the site down |
| Database | SQLite (dev) → PostgreSQL (prod), via SQLAlchemy 2.0 + Alembic | Same ORM code both places; Postgres needed later for full-text search & concurrency |
| Frontend | React 18 + Vite + TypeScript, Tailwind CSS | Fast dev loop; SPA is fine since SEO is a v2 concern |
| API contract | REST + OpenAPI (auto-generated by FastAPI) | Frontend generates its client types from the schema |

## 3. Data Sources (v1 — all free)

Each source is implemented as a **connector**: a Python module with a common interface `fetch() -> list[RawEvent]`.

| # | Source | Method | Notes |
|---|---|---|---|
| S1 | Town government calendars (23 towns: Stamford, Bridgeport, Norwalk, Danbury, Greenwich, Fairfield, Westport, Stratford, Trumbull, Shelton, Darien, New Canaan, Ridgefield, Wilton, Weston, Easton, Monroe, Newtown, Bethel, Brookfield, New Fairfield, Sherman, Redding) | iCal / RSS feeds where available; HTML scrape fallback per town | Feed URL discovery is a one-time manual task tracked in `sources.yaml` |
| S2 | Library calendars | LibCal RSS/iCal feeds (most CT libraries use Springshare LibCal) | Dense source of small community events |
| S3 | CTvisit events (ctvisit.com/events) | HTML scrape, filtered to Fairfield County towns | State tourism: festivals, markets |
| S4 | Patch town calendars | HTML scrape per town page | |
| S5 | HamletHub events | HTML scrape | Coverage may be spotty; degrade gracefully |
| S6 | Experience Fairfield / Visit Fairfield County | HTML scrape | |
| S7 | USDA Local Food Directories | Official free API | Farmers markets (recurring schedules → expand into weekly event instances for the next 8 weeks) |
| S8 | User submissions | Web form → moderation queue | See §6 |

**Connector rules:**
- Every connector declares `source_id`, `fetch()`, and is registered in `sources.yaml` (URL, town, enabled flag, poll frequency).
- Failures are logged per-source and never abort the whole ingestion run.
- Respect robots.txt and set a descriptive User-Agent; per-domain rate limit ≥ 2s between requests.
- Scrapers must be tested against saved HTML fixtures (see §9) so site redesigns are caught by CI, not in prod.

## 4. Data Model

```
Event
  id              uuid pk
  title           text, required
  description     text
  starts_at       timestamptz, required
  ends_at         timestamptz, nullable
  all_day         bool
  venue_name      text
  address         text
  town            text          -- one of the 23 towns, or "Other"
  lat, lon        float, nullable
  url             text          -- canonical/original event page
  image_url       text, nullable
  price_text      text, nullable  -- free-form ("Free", "$10", "Donation")
  status          enum: pending | approved | rejected | expired
  created_at, updated_at

EventSource        -- many-to-one: one Event, many sightings (dedup)
  id, event_id fk
  source_id       text  (e.g. "town-westport", "libcal-fairfield", "user-submission")
  external_id     text  -- id/URL at origin
  raw             jsonb -- original payload for reprocessing
  first_seen_at, last_seen_at
  unique(source_id, external_id)

IngestRun          -- observability
  id, source_id, started_at, finished_at, ok bool, events_found int, error text
```

- Ingested events are auto-`approved`; only user submissions start as `pending`.
- Events with `starts_at` > 60 days out are still stored; UI defaults to the next 30 days.
- Nightly job marks past events `expired` (kept, not deleted).

## 5. Deduplication

Runs at ingest time, before insert:

1. **Exact match:** same `(source_id, external_id)` → update existing (refresh `last_seen_at`, apply field changes).
2. **Cross-source fuzzy match:** candidate = existing event where `starts_at` within ±30 min AND same `town`, then `token_set_ratio(title_a, title_b) ≥ 85` (rapidfuzz). Match → attach as new `EventSource` row to the existing Event instead of creating a duplicate.
3. **Field precedence on merge:** longer description wins; town-government source wins for time/venue; earliest-seen source keeps canonical `url`.
4. Ambiguous matches (score 70–85) are flagged for the admin dedup review queue (v1: logged only; UI in v1.1).

## 6. Event Submission & Moderation

- Public form: title, date/time, venue, town (dropdown of 23), description, link, optional image URL, submitter email.
- Anti-spam: honeypot field + per-IP rate limit (5/day) + required email. No CAPTCHA in v1.
- Submission creates an Event with `status=pending`.
- **Admin UI** at `/admin` behind a single shared password (env var, session cookie). Lists pending events with Approve / Reject / Edit-then-approve. No user account system in v1.

## 7. API (FastAPI, `/api/v1`)

| Endpoint | Description |
|---|---|
| `GET /events` | List approved, upcoming events. Filters: `town`, `from`, `to`, `q` (text search), `free_only`. Paginated (cursor). |
| `GET /events/{id}` | Event detail incl. sources ("Seen on: Westport Town Calendar, CTvisit"). |
| `POST /submissions` | Create pending event (rate-limited). |
| `GET /towns` | Towns + upcoming event counts. |
| `GET /events.ics` | iCal feed of approved events (same filters) — lets users subscribe in Google/Apple Calendar. |
| Admin: `GET/POST /admin/pending`, `POST /admin/events/{id}/approve|reject` | Password-protected. |

## 8. Frontend (pages)

1. **Home / Calendar** — month grid + list toggle; filters: town multi-select, date range, text search. Mobile-first (list default on small screens).
2. **Event detail** — full info, map link, "Add to my calendar" (.ics download / Google Calendar link), original source link.
3. **Submit an event** — the moderation-queue form.
4. **Admin** — pending queue (approve/reject/edit), last ingest run status per source.

## 9. Quality & Testing (spec-driven)

- Each connector ships with ≥ 1 saved HTML/iCal fixture and a parser unit test asserting field extraction.
- Dedup: unit tests covering exact, fuzzy-match, and no-match cases.
- API: pytest + httpx integration tests per endpoint (incl. auth on admin routes, rate limit on submissions).
- CI target: `make test` runs everything; scaffold includes GitHub Actions workflow.
